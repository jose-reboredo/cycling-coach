# Post-mortem — v9.6.4 RSVP rate-limit shape mismatch

## Timeline (UTC, 2026-04-30)

- ~17:30 — v9.6.2 deployed (Sprint 4 Phase 2 — Members tab + RSVP wiring). Smoke probed `POST /rsvp` for 200 OK; did not assert that the count persisted on a follow-up `GET /overview`.
- ~18:30 — User report: "RSVP count shows 1 then suddenly 0 attenders".
- ~18:45 — v9.6.3 hotfix attempted: added `LEFT JOIN event_rsvps` in `GET /overview` so confirmed_count comes from the table instead of a hardcoded `0`. Technically correct change — but wrong root cause for this bug.
- ~19:00 — User: "v9.6.3 LIVE no la veo en prod, los 3 tickets fallan".
- ~19:15 — D1 query: `SELECT COUNT(*) FROM event_rsvps;` returned `0` across all time. Not a single RSVP had ever persisted. The LEFT JOIN was correct but moot — there was nothing to join to.
- ~19:30 — Root cause identified at `src/worker.js:813` (POST `/rsvp`) and `src/worker.js:909` (PATCH `/api/users/me/profile`): `if (!rl.ok)` checking the rate-limit return value, but `checkRateLimit` (added Sprint 1 `#33`) returns `null` (proceed) or `{retryAfter}` (over limit) — never an `{ok}` shape. `null.ok` threw `TypeError`, every first `POST /rsvp` returned 500. Frontend's optimistic-revert kicked in on the error → "shows 1 then snaps back to 0".
- ~19:45 — v9.6.4 shipped (commit `6aa70ab`) with both call sites fixed.

## User-visible symptom

Clicking RSVP increments `confirmed_count` to 1 in the UI, then snaps back to 0 within 1–2 seconds when the next refetch runs. No RSVP persists. Profile-visibility PATCH silently 500'd too — same bug pattern.

## Root cause (`src/worker.js:813` and `:909`)

Phase 2 added two new rate-limited endpoints using the `checkRateLimit` helper. The helper returns:

- `null` when under threshold (proceed)
- `{retryAfter: N}` when over threshold (return 429)

Phase 2 wrote `if (!rl.ok)`. `rl.ok` doesn't exist in either case. When `rl === null`, accessing `null.ok` throws `TypeError`. The Worker handler had no surrounding `try`-catch, so every first `POST /rsvp` and every `PATCH /api/users/me/profile` returned 500 with no error body.

The four other `checkRateLimit` callers in the codebase (`/coach`, `/coach-ride`, `/discover`, the existing `/clubs*` writes) all use the correct `if (rl)` pattern — the bug was isolated to the two new Phase 2 endpoints.

## Fix

Both call sites changed to:

```js
const rl = await checkRateLimit(env, 'clubs-write', String(athleteId), 30, 60);
if (rl) {
  return new Response(JSON.stringify({ error: 'rate-limited', retry_after_seconds: rl.retryAfter }), {
    status: 429,
    headers: { ...corsHeaders, 'Content-Type': 'application/json', 'Retry-After': String(rl.retryAfter) },
  });
}
```

Response shape aligned to the rest of the codebase (`{error: 'rate-limited', retry_after_seconds}`).

## Prevention rule

**Smoke what changed — UPSERT + count-refetch features need POST → GET round-trip smoke, not just 200 OK on POST.** Promoted to `0-learnings.md` Rule #5 (refined from Sprint 1 Improvement #5 + Sprint 4 Improvement #4). Single 200 OK on the POST is theatre when the regression risk is in the persistence layer.

## Smoke addition to `RELEASE_CHECKLIST.md`

Add to the per-release smoke gate (for any release introducing a new POST endpoint with optimistic UI):

- POST the new endpoint with a valid payload → wait 1 second → GET the surface that reads back the persisted state → assert the state matches the POST. Block deploy on regression.
- For RSVP/UPSERT-shape endpoints specifically: `SELECT COUNT(*) FROM <table>;` via `wrangler d1 execute` after a smoke POST should be ≥ 1.
