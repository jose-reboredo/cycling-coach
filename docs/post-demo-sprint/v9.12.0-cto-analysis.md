## §1. Definition

v9.12.0 evolves the v9.11.0 Personal Scheduler from a **club-event aggregator** (events I RSVP'd to + I created) into a **unified training calendar** that also includes **personal training sessions**. It addresses 4 founder-identified gaps in v9.11.0:

1. Personal training sessions are invisible (the AI Coach generates plans but they live in `ai_reports.plan_json` which isn't reliably parseable for calendar rendering).
2. The drawer in `/dashboard/schedule` is read-only — users can't unsubscribe from a club event they RSVP'd to, can't cancel/edit events they created, can't manage their own sessions.
3. There's no visual distinction between event sources (which club, or "this is my training session").
4. A pre-existing CSS bug: TopTabs row leaves empty space on the right because tabs aren't flex-distributed across the available width.

Bundled together because all four touch the same surface (the `/dashboard/schedule` route + `EventDetailDrawer`) and the architectural foundation (new `planned_sessions` table) is the prerequisite for items 1, 2, and 3. Item 4 is a 10-minute CSS fix that ships in the same release.

### What v9.12.0 is NOT

- Not AI Coach integration (i.e. AI-generated plans auto-populating `planned_sessions`). That's `#79`, deferred to v9.13.0+.
- Not goal-anchored scheduling (target_event_date from `goals` table). Deferred until `#49` / `#50` Sprint 7 ships the goals data model.
- Not cross-club admin visibility (a club admin viewing other admins' events). Out of scope; per-club model preserved.

## §3. Architecture

### §3.1 Data layer — Migration 0008

New table `planned_sessions` (additive; no existing columns touched, no breaking change):

```sql
CREATE TABLE planned_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  athlete_id INTEGER NOT NULL REFERENCES users(athlete_id) ON DELETE CASCADE,
  session_date INTEGER NOT NULL,                  -- unix epoch seconds (start time)
  title TEXT NOT NULL,                            -- e.g. "Sweet-spot intervals · 3×12"
  description TEXT,                               -- optional notes / coach guidance
  zone INTEGER CHECK (zone IS NULL OR zone BETWEEN 1 AND 7),
  duration_minutes INTEGER CHECK (duration_minutes IS NULL OR duration_minutes BETWEEN 0 AND 600),
  target_watts INTEGER CHECK (target_watts IS NULL OR target_watts BETWEEN 0 AND 2000),
  source TEXT NOT NULL DEFAULT 'manual'
    CHECK (source IN ('manual', 'ai-coach', 'imported')),
  ai_report_id INTEGER REFERENCES ai_reports(id) ON DELETE SET NULL,
  completed_at INTEGER,                           -- soft completion (nullable)
  cancelled_at INTEGER,                           -- soft delete (nullable)
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE INDEX idx_planned_sessions_athlete_date
  ON planned_sessions(athlete_id, session_date);

CREATE INDEX idx_planned_sessions_ai_report
  ON planned_sessions(ai_report_id)
  WHERE ai_report_id IS NOT NULL;
```

**Design choices and why:**

- **`athlete_id` not `user_id`**: matches existing convention (every other table uses `athlete_id` for FK to `users.athlete_id`).
- **`session_date INTEGER` (epoch)**: same convention as `club_events.event_date`. Avoids timezone ambiguity; all displayed times are derived in the client from `Intl.DateTimeFormat`.
- **`zone INTEGER` not TEXT**: enforces 1-7 Coggan range; integer math for color-coding.
- **`source` enum**: enables analytics — what fraction of sessions are AI-generated vs manual vs imported (Strava planned activities). v9.12.0 ships `manual` only; `ai-coach` lands with `#79`.
- **`ai_report_id` FK with `ON DELETE SET NULL`**: when a stale AI report is purged (future cleanup), the planned session survives but loses its provenance link. Acceptable.
- **No `cancelled_by`**: only the athlete owns their planned sessions; no admin override. Enforced by endpoint auth gate.
- **Two soft-state columns** (`completed_at`, `cancelled_at`): completion and cancellation are independent — a session can be cancelled before its time, never marked complete. Both nullable, both timestamps for analytics later.
- **`updated_at`**: enables "last modified" sort + simple optimistic-lock pattern if we add concurrent-edit detection later.

**Index choices:**

- Primary index `(athlete_id, session_date)`: covers the dominant query pattern — "give me this athlete's sessions in a date range". Composite index serves both columns; partial scans on date alone wouldn't be needed.
- Partial index `idx_planned_sessions_ai_report ... WHERE ai_report_id IS NOT NULL`: 90% of rows in v9.12.0 will have NULL here (manual creation only); partial index keeps storage cost minimal until AI integration.
- No index on `cancelled_at` — filter is `WHERE cancelled_at IS NULL` which the query planner handles via composite index sort + filter.

### §3.2 API design

Five new endpoints + one extension:

| Verb | Path | Purpose |
|---|---|---|
| GET | `/api/me/sessions?range=YYYY-MM` | List sessions for month (calendar render) |
| POST | `/api/me/sessions` | Create manual session |
| PATCH | `/api/me/sessions/:id` | Edit (title/zone/duration/target_watts/description), mark `completed_at`, etc. |
| POST | `/api/me/sessions/:id/cancel` | Soft-delete (set `cancelled_at`) |
| POST | `/api/me/sessions/:id/uncancel` | Restore (set `cancelled_at = NULL`) |
| GET | `/api/me/schedule?range=YYYY-MM` (extend) | Now returns `{club_events, planned_sessions}` instead of just `events` |

**Auth gates:** all endpoints require `resolveAthleteId(request)`. Sessions endpoints additionally enforce `athlete_id = caller` server-side (no admin override).

**Rate limits:** new `me-sessions-write` scope, 30/min/athlete on POST/PATCH/Cancel/Uncancel. Read endpoints unmetered.

**Response shapes:**

`GET /api/me/sessions?range=2026-05`:
```json
{
  "athlete_id": 12345,
  "range": { "year": 2026, "month": 5, "start": ..., "end": ... },
  "sessions": [
    {
      "id": 1,
      "session_date": 1717142400,
      "title": "Sweet-spot intervals · 3×12",
      "description": "Form is productive — push the engine.",
      "zone": 4,
      "duration_minutes": 75,
      "target_watts": 252,
      "source": "manual",
      "completed_at": null,
      "cancelled_at": null,
      "created_at": 1715000000,
      "updated_at": 1715000000
    }
  ]
}
```

`GET /api/me/schedule?range=2026-05` (extended):
```json
{
  "athlete_id": 12345,
  "range": {...},
  "club_events": [...],          // existing v9.11.0 shape
  "planned_sessions": [...]      // NEW — same shape as /api/me/sessions
}
```

**Idempotency:** PATCH is idempotent (same input → same output). Cancel is idempotent — second call returns the existing `cancelled_at`. Uncancel is idempotent. POST is intentionally non-idempotent (creates new session each time); client-side dedup not required for v9.12.0 because users explicitly tap "Add session".

**Error responses:** standard `{error: 'message'}` shape, status codes:
- 400: validation errors (title missing, zone out of range, duration > 10h, etc.)
- 401: unauthenticated
- 403: trying to edit another user's session
- 404: session not found OR not yours (OWASP — don't leak existence)
- 429: rate-limited
- 500: D1 failure (with `safeWarn` log)

## §4. Scalability analysis

Target: 1000 active users; comfortable headroom to 10k.

| Vector | Math | Verdict |
|---|---|---|
| Storage | 1k users × 25 sessions/month × ~200B/row = 5 MB/month, ~60 MB/year. Indexes ~30% overhead → ~80 MB/year. D1 free tier: 5 GB. | ✅ 60+ years of headroom on free tier |
| Read latency (single-user month range) | Index seek `(athlete_id, date_range)` returns ~25 rows. D1 query ~5-10ms cold, sub-ms hot. + 5-min edge cache fronts most requests. | ✅ Well under 100ms p99 target |
| Write throughput | 1k users × 10 writes/week (avg) = 10k writes/week = 1.5/sec peak. D1 sustained: 100s/sec. | ✅ 60× headroom |
| Aggregation `/api/me/schedule` | 1 query joining `club_events + clubs + club_members + event_rsvps + planned_sessions`. With proper indexes ~10-20ms cold. 5-min edge cache absorbs bursts. | ✅ |
| Concurrent users | Cloudflare Workers handle ~10k concurrent requests/instance; D1 connection pool managed. No per-user state in worker. | ✅ |
| Memory per user (client) | ~25 sessions × ~500B JSON = 12 KB / month / user. Tanstack Query cache holds ~3 months = 36 KB. | ✅ Negligible |

**Failure modes and degradation:**

- **D1 timeout (>5s)**: returns 500; client falls back to last-cached (Tanstack Query 5-min stale). User sees yesterday's data, not blank screen.
- **Edge cache stampede on cold-start**: 5-min cache absorbs N-1 of N concurrent requests; only first hits D1.
- **Athlete with thousands of sessions** (power user, multi-year history): query targets month range only; no user-level scan. O(M) where M = sessions in range, not total.

**When does this need rework?** Conservatively, around **50-100k users** the aggregation query joining 5 tables for cross-club events may need:
- Materialized view of "upcoming events per athlete" rebuilt nightly via cron
- KV-backed read-through cache keyed by `athlete_id:range`

Both are evolutionary, not breaking. Migration path documented; no need to pre-build.

## §5. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| **R1: Migration 0008 fails on remote D1** | Low | High (deploy blocked) | Apply locally first + verify via `PRAGMA table_info`; remote apply is single ALTER, atomic; rollback = `DROP TABLE planned_sessions` (no FK cascade since no other table references it yet) |
| **R2: Existing tests break** (TopTabs flex-distribute changes layout) | Medium | Low | Run full e2e locally before deploy; visual regression caught on real-device smoke (RELEASE_CHECKLIST iPhone gate) |
| **R3: 5-min edge cache serves stale data after Cancel/Edit** | High | Medium | Tanstack Query invalidation on mutations clears client cache; edge cache resolves naturally within 5 min. Document the up-to-5-min staleness in user-facing copy if needed; for ride-day decisions ≤ 5 min lag is acceptable |
| **R4: Permission gate bypass** (user edits another user's session) | Low | High (data-integrity) | Server-side `WHERE athlete_id = ?` on every PATCH/Cancel/Uncancel; tested via e2e auth-gate suite. Client-side gating is UX-only, not security |
| **R5: Performance regression at 10k users from N+1 in `/api/me/schedule`** | Medium | Medium | Single consolidated query + composite indexes; no N+1 by design. Load test with 10k synthetic athletes before crossing that threshold |
| **R6: Cancelled events still appearing on calendar (regression of `#74`)** | Medium | Low | Test case in e2e: cancel event → assert it's gone from upcoming list, present with strikethrough on grid. Same for sessions |
| **R7: Visual-differentiation palette clashes with v9.7.0+ event-type colors** | Low | Low | Personal sessions use zone palette (already in tokens); club events keep current event-type colors. No overlap |
| **R8: Data race when same session is edited from two devices** | Low | Low | Last-write-wins (current pattern). `updated_at` enables future optimistic-lock if needed; not blocking |
| **R9: Soft-delete confusion** — user expects "Cancel" to fully remove the session; sees it returning via Uncancel UX | Low | Low | Don't ship Uncancel UX in v9.12.0; backend supports it but no UI button. Hard delete via separate endpoint if requested later |
| **R10: AI Coach integration (`#79`) creates duplicate sessions** when re-running `/coach` | High (when shipped) | Medium | Future: dedup on `(athlete_id, session_date, ai_report_id)` unique constraint OR upsert pattern. v9.12.0 doesn't ship this; add to `#79` spec |

### §7.4 Rollback path

**Code rollback** (if frontend or worker bug): `git revert <release-commit>` + `npm run deploy`. Migration 0008 stays — additive table doesn't affect older code paths.

**Data rollback** (if data corruption): `DROP TABLE planned_sessions;` via wrangler d1 execute. Loses any user-created sessions but doesn't affect club events or other tables.

**Partial rollback**: revert just the worker (keeps migration); endpoints return 404; frontend gracefully renders empty `planned_sessions: []`. No user-facing crash.

### §7.5 Observability

Existing patterns (no new infrastructure):

- `safeWarn` log for D1 failures, parse errors, Anthropic key issues
- Cloudflare Workers analytics for request volumes / latencies on the new endpoints
- Manual D1 query for `SELECT COUNT(*) FROM planned_sessions` to track adoption

Future (not v9.12.0): add a `/admin/stats` endpoint that returns daily creation count, source distribution, completion rate.

## §9. Open questions / decisions to lock

1. **Add Session pattern: page route or modal?** Recommendation: **page route** at `/dashboard/schedule/new` — consistent with v9.8.2 architectural lesson (Rule #17: pattern-replacement when modal proves fragile). One less modal in the codebase.

2. **Visual-differentiation between clubs**: per-club deterministic palette color or just club-name chip on the pill? Recommendation: **club-name chip only** for v9.12.0; per-club color is v9.13.0+ polish. (Already locked from prior decisions, restating for completeness.)

3. **Drawer split: one EventDetailDrawer with `event_source` branching, or two separate drawers (`EventDetailDrawer` + `SessionDetailPanel`)?** Recommendation: **branch in EventDetailDrawer** for v9.12.0 — keeps single drawer entry-point, single set of motion animations, single z-index/portal handling. Refactor to two components only if EventDetailDrawer exceeds ~300 lines.

4. **AI Coach integration (`#79`)** — when does it ship? Recommendation: **v9.13.0**, after v9.12.0 manual sessions prove the data model. Don't bundle.

5. **Hard delete vs soft delete** for personal sessions: recommendation **soft delete only** (set `cancelled_at`). User-facing UX shows "Delete" button which calls `/cancel`. Hard delete only via DBA tooling. Audit trail preserved.

6. **Multi-device concurrent edit** — last-write-wins acceptable? Recommendation: **yes**, for v9.12.0. `updated_at` column enables optimistic-lock if needed later (v9.14.0+ when teams + sharing land).
