# v8.5.0–v8.5.3 backlog burn — design doc

**Date:** 2026-04-28 · **Author:** Claude (per `superpowers:brainstorming`) · **Approved by:** José Reboredo (option-(a) approval, this conversation)

> Output of a multi-turn brainstorm: shape **B** (milestone-by-milestone with tests added per issue) at scope **v8.5.0 milestone** (~14 issues), per-group cadence (4 releases), pragmatic test depth.

## Context

`/roadmap` reports 27 open issues across milestones v8.3.0 / v8.4.0 / v8.5.0 / unmilestoned. Repo has zero regression tests today — `npm run build:web` (TS strict + Vite build) and Biome lint are the only automated gates. Manual smoke at 375 + 1280 px before each deploy.

The user wants to drain the **v8.5.0 milestone** specifically (14 open issues), with **regression tests added per release**.

## Scope

### In scope

- Implementing all open v8.5.0 issues (12 net, after triage)
- Building a pragmatic regression-test harness (Playwright e2e smoke + Vitest unit tests on touched libs + GitHub Actions gating deploy)
- Four release cuts: v8.5.0 (Polish · 5 issues) → v8.5.1 (Security · 5) → v8.5.2 (D1 strangler · 1) → v8.5.3 (Lighthouse · 1)
- Each release: build green · both test suites green · CI green · manual smoke · `npm run deploy` · Confluence sync · CHANGELOG + README + version bump

### Out of scope

- Issues outside the v8.5.0 milestone (v8.6.0 backlog, "Someday" items)
- Full e2e coverage of every dashboard surface (smoke only — Phase 0 covers 3 critical paths)
- Rebuilding any prior milestone (v8.3.0 / v8.4.0 / v8.4.1 already shipped)
- Design-system overhaul (PARS stays — molten orange + Geist + cockpit dark)
- Backfill TSS work (separate issue, already deferred)

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Cloudflare Rate Limiting binding (#18) requires account-tier config beyond the Worker manifest | If the binding can't be created via `wrangler.jsonc` alone, defer #18 to v8.5.4 and ship the rest of Phase 2 |
| Phase 3 (D1) blocks until the user runs the remote migration | Phase boundary explicitly pauses for confirmation; not in-flight state |
| Playwright in CI flakes on `wrangler dev` startup timing | `webServer.command` with generous `waitFor: 'networkidle'`, 30s timeout, retries: 1 |
| 12 issues × full SDLC = many small back-and-forths | For S-effort items (#19, #20, #22, #25) I batch the design check into the release plan; full per-issue brainstorm reserved for M+ items |
| Self-hosting Geist (Phase 4) changes how fonts load — could regress visual rendering | Lighthouse is downstream verification; extra manual smoke at 375 + 1280 before deploy |
| Issue scope drift mid-phase | At each phase start I re-confirm scope ("starting Phase 2 — same 5 issues, OK?"); skill-locked: each phase is independently shippable, pause anywhere |

## Open questions / assumptions to revisit

- **Issue closing path** — committed to the `/admin/close-issues` Worker route (see Phase 0a). Open question retired.
- **Phase 3 timing** — depends entirely on when the user runs the D1 migration. If they want it now, Phase 3 happens before Phase 2; if they prefer to defer, Phase 3 slips and we close out v8.5.0 + v8.5.1 first.
- **Phase 4 Lighthouse target** — "≥ 90" is the brief; if `/dashboard` proves stubbornly < 90 due to first-load bundle weight (Motion + Tanstack), we accept ≥ 85 with a documented "verified-good user-perceived performance" note rather than ship a thrashed bundle just to chase the number. Decision deferred to that phase.
- **Per-issue branch strategy** — current pattern is direct-to-main commits (no feature branches). Continuing with that is fine for solo dev; if the user wants a `feat/v8.5.x-burn` branch + final merge, that's a Phase 0 decision.
